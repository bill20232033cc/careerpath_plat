## 📌 产品核心能力（7大模块）

| 模块 | 核心能力 | 关键页面/接口 |
|------|----------|--------------|
| 1. 简历解读 | 上传/粘贴简历，AI 解读关键需求、竞争力、短板 | `/resume`, `POST /api/resume` |
| 2. 岗位匹配 | 根据简历分析 + 求职意向，搜集主流招聘平台信息 | `/jobs`, `POST /api/jobs/match` |
| 3. 目标确认 | 整理匹配岗位，圈定目标，交给求职者确认 | `/jobs`, `/dashboard` |
| 4. 技能拆解 | 将岗位需求拆解为技能模块，Flipbook 可视化呈现（含 ASCII 艺术化） | `/path`, `POST /api/path/generate`, `SkillFlipbook` |
| 5. 优化建议 | 个性化简历优化建议、资格考试方向、在线课程推荐、实习情报 | `/learning`, `GET /api/learning` |
| 6. 社区激励 | 社区交流 + 成就激励系统 | `/community`, `/achievements`, `POST /api/community/posts` |
| 7. 用户系统 | 注册登录功能 | `/login`, `/register`, `POST /api/auth/login`, `POST /api/auth/register` |

### 核心用户旅程
```
注册登录 → 上传/粘贴简历 → AI 解读简历 → 匹配岗位 → 确认目标 → 
技能拆解(Flipbook) → 优化建议/课程推荐 → 社区交流 → 成就激励
```

## 🚀 强制全流程执行规范

### 阶段1：前置锚定 - 需求→设计→架构对齐校验
用户输入核心信息后，必须先完成本阶段，绝对不能直接写代码：
1. **需求拆解**：将自然语言需求拆解为结构化 PRD
2. **设计落地**：拆解核心用户旅程、业务流程、页面交互、数据模型
3. **架构对齐**：确认技术栈选型、分层架构、模块划分、数据流走向
4. **输出交付**：《需求/设计/架构对齐文档》，待用户确认后进入下一阶段

### 阶段2：首轮 Vibe Coding - 生产级 MVP 骨架搭建
1. 生成符合架构规范的完整项目骨架
2. 实现核心业务链路的全流程代码
3. 代码必须可直接编译、运行、调试
4. 输出交付：完整项目代码、《项目结构说明》、《本地启动指南》

### 阶段3：多轮迭代循环（每轮完成后输出执行报告）

#### 循环1：需求/设计/架构调整 → 迭代
- 补全细节遗漏或响应用户调整指令
- 仅修改对应模块，同步更新关联逻辑

#### 循环2：代码重构 → 优化
- 消除重复代码、优化命名与分层、提升可读性
- 必须保证功能100%不变，先写测试验证再重构

#### 循环3：Corner Case 边界场景 → 全覆盖
- 覆盖空值、非法入参、数据越界、网络异常、第三方服务不可用等
- 补充防御性代码和单元测试

#### 循环4：容错处理 → 全链路落地
- 统一异常拦截、分级错误码、重试/超时/降级、兜底返回、结构化日志

#### 循环5：高度抽象 → 封装复用
- 抽取通用能力、封装公共组件/工具类、面向接口编程、消除硬编码

#### 循环6：依赖检查 → 合规治理
- 禁止循环依赖、反向依赖、隐式行为依赖
- 调用链路必须符合架构分层规范

#### 循环7：可扩展性 → 提前预留
- 遵循开闭原则，预留扩展点，实现配置化/策略模式设计

### 阶段4：最终交付 - 生产级完整交付
1. 全量可编译、可直接部署的项目代码
2. 全链路测试用例及测试报告
3. 完整项目文档
4. 环境配置文件、部署相关文件

## 🏗️ 技术架构规范

### 技术栈
| 层级 | 技术 | 说明 |
|------|------|------|
| 前端框架 | Next.js 14 (App Router) | React 18, TypeScript |
| 样式方案 | Tailwind CSS + Radix UI | shadcn/ui 组件模式 |
| 动画 | Framer Motion | 页面过渡、Flipbook 交互动画 |
| AI 服务 | DeepSeek API (OpenAI SDK) | 简历分析、岗位匹配、路径生成 |
| 认证 | NextAuth.js | 注册/登录/会话管理 |
| 密码加密 | bcryptjs | 用户密码哈希 |
| 数据存储 | JSON 文件 (lib/data/) | 当前阶段，后续迁移数据库 |

### 分层架构
```
app/                    → 接口层（页面路由 + API 路由）
  (auth)/               → 认证页面组
  (main)/               → 主业务页面组
  api/                  → API 路由层
components/             → 组件层
  ui/                   → 基础 UI 组件（shadcn 风格）
  resume/               → 简历模块组件
  jobs/                 → 岗位模块组件
  flipbook/             → Flipbook 可视化组件
  community/            → 社区模块组件
lib/                    → 通用层
  data/                 → 数据文件（JSON）
  types.ts              → 类型定义
  utils.ts              → 工具函数
  deepseek.ts           → AI 服务封装
```

### 模块边界规范
| 模块 | 负责 | 不负责 |
|------|------|--------|
| 页面组件 (`app/`) | 路由、页面布局、数据获取 | 业务逻辑计算 |
| API 路由 (`app/api/`) | 业务逻辑处理、AI 调用、数据操作 | 前端界面渲染 |
| UI 组件 (`components/ui/`) | 基础交互组件 | 业务逻辑 |
| 业务组件 (`components/resume/` 等) | 模块专属交互与展示 | 跨模块数据操作 |
| 通用层 (`lib/`) | 类型定义、工具函数、AI 封装、数据存取 | 页面渲染 |

### 数据流走向
```
用户操作 → 页面组件 → fetch API 路由 → callDeepSeek / 数据操作 → 返回 JSON → 页面更新
```

## 📚 附录：技术设计指导原则

### 代码编写原则

#### 1. 注释与文档
- 所有函数必须提供完整中文 docstring（参数、返回值、异常）
- 函数内复杂逻辑必须有中文行内注释说明意图
- 注释要说明"为什么这么做"，而不是"做了什么"

#### 2. 代码结构
- 保持函数扁平化，优先可测试性与可维护性
- 模块间依赖最小化，优先接口/协议而非具体实现
- 重复逻辑必须抽取为公共函数/类
- 禁止魔法数字/魔法字符串

#### 3. 实现策略
- 设计前先选择最佳实践方案，直接按最佳实践实现
- 编写处理规则时禁止硬编码规则，要写自适应规则

#### 4. 测试策略
- 每完成一处代码修改，应同步编写/更新测试
- 核心功能必须有单元测试
- 单文件测试覆盖率目标为 ≥80%

#### 5. 前后端一致性
- API 请求/响应的字段名必须与 `lib/types.ts` 中的类型定义一致
- 新增 API 必须先定义类型，再实现接口
- 前端 fetch 的 body 字段名必须与后端 `request.json()` 解析的字段名完全匹配

#### 6. AI 服务调用规范
- 所有 AI 调用统一通过 `lib/deepseek.ts` 的 `callDeepSeek` 函数
- systemPrompt 必须明确指定返回的 JSON 结构，与 `lib/types.ts` 中的类型对齐
- AI 返回结果必须做 JSON.parse 容错处理，解析失败时提供合理的 fallback 数据
- fallback 数据结构必须与正常返回结构完全一致
